# 重构「不做什么」清单（Do-Not List）

> 本清单的目的：保护重构纪律。
> 重构期间，凡是出现在本文档中的行为，**任何 PR 不予合并**。
> 配套决策见 [ADR-0001](../adr/0001-migrate-to-microservices.md)。

## 〇、为什么需要这份清单？

重构最常见的失败模式不是"做不完"，而是"边重构边改东西，改到最后没人知道哪个是 bug、哪个是新设计"。

历史上死在这上面的项目数不胜数。这份清单的存在是为了**给团队一个明确的拒绝理由**：
当业务方、Tech Lead、或自己手痒想"顺手改一下"时，可以指着这份文档说"不行，按规矩来"。

**核心原则**：**重构期间，唯一目标是「让目标架构跑起来」，不是「让代码变更好」。**

## 一、🚫 业务功能层面

### 1.1 不做新功能

| 行为 | 为什么禁止 | 例外 |
|------|----------|------|
| 加新的业务功能 | 重构期已承诺业务方功能冻结 | 仅 P0 bug 修复 |
| 新增 API 接口（除了 proto 重写） | 增加搬迁工作量 | 无 |
| 调整业务规则（如计费逻辑） | 会让"搬迁后行为不一致"无法判断是 bug 还是有意 | 无 |
| 加新的页面/路由 | 同上 | 无 |
| 改 UI 视觉/交互 | 前端重构期间一律保持现状 | 仅修复明显错乱 |

**判断标准**：如果一个改动让"重构前后行为不一致"，它就是新功能，不是重构。

### 1.2 不主动废弃功能

- 即使你觉得某个功能没人用、设计很丑、技术债很重，也**不要在重构期间删除它**
- 保留它，搬迁过去，等重构完成后再用专门的 PR 删除
- **理由**：删除功能 = 业务变更 = 需要业务方确认 = 拖慢重构

## 二、🚫 代码层面

### 2.1 不做"顺手优化"

| 行为 | 为什么禁止 |
|------|----------|
| 重命名变量/函数（因为"原来名字不好"） | 制造大量 diff，掩盖真正的搬迁逻辑 |
| 把 `for` 改成 `range` 或类似的语法糖优化 | 同上 |
| 拆分大函数（因为"太长了"） | 重构期间大函数照搬，等稳定后再拆 |
| 改变错误处理风格 | 同上 |
| 添加注释/文档 | 除非业务逻辑确实需要解释 |
| 调整 import 顺序 | 让 git blame 失效 |
| 引入新的依赖库 | 增加未知风险 |

**唯一允许的"优化"**：改 import 路径以适配新目录结构。

### 2.2 不做架构层面的"创新"

| 行为 | 为什么禁止 |
|------|----------|
| 引入新的设计模式（CQRS、Event Sourcing 等） | ADR 没说 = 不做 |
| 改变 ORM 用法（如从 GORM 切到 sqlc） | ADR 已明确「技术栈维持现状」 |
| 引入服务网格（Istio/Linkerd） | 阶段 2 之后再讨论 |
| 引入分布式追踪以外的复杂中间件 | 同上 |
| 把 REST handler 全删了直接上 gRPC | REST 适配层必须保留作为兼容缓冲 |

### 2.3 不混合多种类型的变更

**一个 PR 只做一件事**：

| ❌ 禁止 | ✅ 应该 |
|--------|--------|
| 「搬迁 device 模块 + 修复一个 bug + 重命名 3 个变量」 | 拆成 3 个 PR |
| 「搬迁 + 加测试 + 改 schema」 | 搬迁先做，测试和 schema 用单独 PR |

**判断标准**：如果 PR 标题需要用"和"连接两件事，就该拆开。

## 三、🚫 数据库层面

### 3.1 不在搬迁阶段改数据库

虽然 ADR 允许"数据库 schema 重设计"，但**不能和业务搬迁混在一起**：

```
正确的顺序：
  1. 阶段 0：搭骨架（不动数据）
  2. 阶段 1：搬业务（不动数据库结构，原样搬）
  3. 阶段 1 末期 / 阶段 2 初：用单独 PR 做 schema 重设计
```

**原因**：搬迁时如果同时改 schema，一旦出 bug 你无法判断是搬迁逻辑错了还是 schema 改错了。

### 3.2 不绕过 ORM 中间件

| 行为 | 为什么禁止 |
|------|----------|
| 直接用 `db.Raw(...)` 跳过 `WithTenant()` | 破坏租户隔离 |
| 直接用 `db.Exec(...)` 跳过 `WithTenant()` | 同上 |
| 在 Repository 外部访问 `*gorm.DB` | 同上 |
| 自己拼接 SQL 字符串 | 同上 |

**例外**：极少数场景必须用原生 SQL（如复杂的报表聚合），必须：
1. 在代码里加注释 `//nolint:tenant-bypass reason=xxx`
2. 在 PR 描述中说明理由
3. 必须有 reviewer 显式批准

### 3.3 不做跨 database 的 JOIN

```sql
-- ❌ 禁止
SELECT t.*, p.*
FROM parkhub_core.tenants t
JOIN parkhub_billing.bills b ON b.tenant_id = t.id;

-- ✅ 正确
-- 在应用层：
--   1. 调用 core.TenantService 拿到租户
--   2. 调用 billing.BillService 拿到账单
--   3. 在 BFF 层组装
```

**原因**：未来 MySQL 物理拆库时，跨库 JOIN 会全部失效。

## 四、🚫 Proto 层面

### 4.1 不做 Breaking Change

一旦 proto 定义合并到 `main`，**禁止**：

| 行为 | 替代方案 |
|------|---------|
| 删除字段 | 用 `reserved` 保留 number |
| 修改字段 number | 加新字段 |
| 修改字段类型 | 加新字段（不同名） |
| 删除 RPC 方法 | 标记 `deprecated`，等下一个大版本 |
| 修改 RPC 方法签名 | 加新方法 |

**强制工具**：CI 必须运行 `buf breaking`，发现 breaking change 直接挡 PR。

### 4.2 不跨 domain import proto

```protobuf
// ❌ 禁止
import "billing/v1/rule.proto";  // 在 event/v1/transit.proto 里

// ✅ 正确
// 共享类型放到 common/v1/，所有 domain 都可以引用
import "common/v1/money.proto";
```

## 五、🚫 流程层面

### 5.1 不直接合并到主分支

重构期间所有改动走 **`refactor/microservices` 长期分支**：

```
main                            ← 老单体，只接受 P0 bug 修复
  │
  └─ refactor/microservices     ← 重构主分支
       ├─ refactor/phase-0/skeleton
       ├─ refactor/phase-1/core
       ├─ refactor/phase-1/iot
       └─ ...
```

切换日才把 `refactor/microservices` 合并/替换到 `main`。

### 5.2 不省略 Code Review

- 重构 PR 必须 **2 个 reviewer**（其中 1 个是架构组成员）
- 不允许"小改动直接合"
- 即便是搬迁类 PR 也必须 review，因为搬迁错误的代价比新写更大

### 5.3 不省略测试

- 每个 domain 搬迁完成后**必须有单元测试**
- 必须包含**租户隔离测试用例**
- 必须有**端到端集成测试**（至少覆盖关键路径）

### 5.4 不允许"我本地能跑就行"

- 所有 PR 必须通过 CI
- CI 必须包含：`buf lint` + `buf breaking` + `go test` + 自定义 linter（租户绕过检查）
- 红色 CI 不能合并，**禁止用 "rerun" 当作绿色判定标准**

## 六、🚫 沟通层面

### 6.1 不接受口头需求

任何业务方在重构期间提出的需求：
- 必须走 issue 进入"重构后队列"
- 不允许"就这一次帮我加一下"
- Tech Lead / PM 是这道防线的执行者

### 6.2 不在站会上讨论"要不要回头改 X"

站会只讨论：
1. 本次搬迁的进度
2. 遇到的阻塞
3. 下一步搬迁的目标

如果有人提出"我们顺便把 X 也改了吧"，PM/Tech Lead 应该回答：**「记下来，重构完再说」**。

### 6.3 不允许「重构疲劳期」破坏纪律

重构第 4-6 周左右，团队大概率会进入低谷期，出现的论调包括：
- "感觉这样拆没什么用啊"
- "要不就停在阶段 1 算了"
- "我顺手把 X 也改了，反正都在重构"

**应对**：
- ADR 文档作为定海神针
- 每个阶段验收作为里程碑庆祝
- 允许停在阶段 1（Modular Monolith），但**不允许半途变更纪律**

## 七、✅ 允许的例外

为了避免清单变成教条，明确允许以下情况：

1. **P0 生产 bug 修复**：在 `main` 分支上修，然后 cherry-pick 到 `refactor/microservices`
2. **阻塞重构进度的依赖问题**：如发现某个第三方库不兼容，必须升级时
3. **安全漏洞修复**：CVE 级别的问题立即处理
4. **CI 基础设施修复**：CI 红了必须修

**所有例外必须**：
- 在 PR 描述中明确说明"这是清单允许的例外"
- 由架构组成员显式批准
- 记录到重构周报中

## 八、🆘 当你想破例时怎么办

如果你在重构过程中产生了"我想顺手做 X"的冲动：

1. **先停下来 5 分钟**
2. 问自己：「这件事不在重构后做会死人吗？」
3. 如果答案是"不会"，就**记到 issue 里**，等重构完做
4. 如果答案是"会"，找架构组讨论
5. **绝对不要直接动手**

**记住**：重构期间，你的耐心是这个项目最稀缺的资源。

## 九、清单的清单（TL;DR）

如果只能记住 7 件事，记这些：

1. ❌ 不做新功能
2. ❌ 不顺手优化代码
3. ❌ 不在搬迁阶段改数据库 schema
4. ❌ 不绕过 ORM 中间件
5. ❌ 不做 proto breaking change
6. ❌ 不省略 review 和测试
7. ❌ 不接受口头需求
