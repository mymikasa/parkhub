## Context

设备管理基础能力已存在于 `parkhub-api` 和 `parkhub-web`，包括设备列表、详情、统计、改名和心跳驱动的状态更新，但设备仍缺少正式的绑定/解绑工作流。当前代码里设备关联同时出现在 `devices.parking_lot_id/gate_id` 与 `gates.device_id` 两处，其中 `gates.device_id` 延续的是“一个出入口绑定一台设备”的旧模型，已与 Issue #17 的“一个出入口最多 3 台设备”冲突。

前端设备管理页已经具备设备卡片、基础筛选和编辑名称能力，租户与车场也分别已有查询接口和 hooks。真正的缺口是：

- 后端没有 bind/unbind API，也没有容量、状态和租户归属校验。
- 平台管理员在设备页选择租户后，缺少稳定的“按租户加载车场和出入口”的方案。
- 出入口能力当前仍把设备绑定视为 gate 侧行为，不适合作为多设备挂载的长期模型。

## Goals / Non-Goals

**Goals:**

- 为管理员提供可测试的设备绑定与解绑 API，并满足 Issue #17 的权限、状态和容量约束。
- 让设备管理页支持一步完成租户、车场、出入口选择，并可对已绑定设备执行解绑或重新分配。
- 将设备与出入口的关系统一到设备侧数据模型，避免继续扩大 `gates.device_id` 的单设备语义。
- 调整现有出入口规范，使其不再要求在出入口创建/编辑流程中直接绑定单台设备。

**Non-Goals:**

- 不在本变更内完成 MQTT、控制指令、告警、批量绑定等后续设备能力。
- 不删除 `gates.device_id` 数据列；该列在本期只降级为遗留兼容字段。
- 不重做整个出入口管理 UI，只做必要的规范对齐和读路径调整。

## Decisions

### 1. 以 `devices` 表作为绑定真源

绑定关系以 `devices.tenant_id`、`devices.parking_lot_id`、`devices.gate_id` 为准，bind/unbind 只更新设备记录。这样可以直接表达“多个设备归属同一出入口”，不需要引入新的关联表，也不需要沿用 `gates.device_id` 的单值限制。

备选方案：

- 继续把 `gates.device_id` 当真源并额外维护副表。该方案会让读写路径双写，且很难和现有单字段模型兼容。
- 立即新建 `gate_devices` 关联表。该方案最规范，但超出 Issue #17 所需改动量，也会引入额外迁移和读写复杂度。

### 2. bind API 支持对 `pending` 和 `active` 设备直接绑定

`POST /api/v1/devices/:id/bind` 允许绑定待分配设备，也允许把已绑定设备直接改绑到其他出入口。服务层在一个事务里完成以下步骤：

- 校验操作者角色，仅 `platform_admin` 和 `tenant_admin` 可调用。
- 读取设备并校验当前状态只能是 `pending` 或 `active`。
- 校验目标租户、车场、出入口存在且层级归属一致。
- 校验租户权限：
  - `platform_admin` 可指定任意有效租户。
  - `tenant_admin` 只能绑定到自身租户。
- 统计目标出入口当前已绑定设备数，达到 3 台则拒绝。
- 更新设备的 `tenant_id`、`parking_lot_id`、`gate_id`，并把状态置为 `active`。

这样可以覆盖“分配 pending 设备”和“重新分配 active 设备”两类场景，避免前端必须先发 unbind 再发 bind 的双请求流程。

备选方案：

- 强制先解绑再绑定。该方案更贴近旧 PRD 描述，但会让前端交互变成两步提交，也更容易在中间态失败。

### 3. unbind 恢复到平台待分配池

`POST /api/v1/devices/:id/unbind` 会清空 `parking_lot_id` 和 `gate_id`，同时把 `tenant_id` 置回 `tenant-platform`，并把设备状态设为 `pending`。这样能和现有设备状态机保持一致：

- 未分配设备属于平台待分配池。
- 心跳恢复时，未分配设备会从 `offline` 回到 `pending`，已分配设备才会回到 `active`。

备选方案：

- 解绑后保留原租户与 `active` 状态。该方案会使“未绑定但 active”成为合法状态，和当前状态语义冲突，也会让 pending 设备可见性规则变得模糊。

### 4. 平台管理员的绑定选项通过扩展现有车场列表过滤实现

前端绑定弹窗按以下方式加载选项：

- `platform_admin` 先调用租户列表接口选择租户。
- 车场列表复用 `GET /api/v1/parking-lots`，增加可选 `tenant_id` 过滤参数，仅平台管理员可用。
- 出入口列表继续复用 `GET /api/v1/parking-lots/:id/gates`。
- `tenant_admin` 跳过租户选择，直接使用当前租户加载车场和出入口。

选择扩展现有车场列表，而不是新增专用 bind-options 接口，原因是现有前端已具备 tenant/parking-lot/gate 查询链路，增量最小，也更容易复用缓存和现有 hooks。

### 5. 出入口读路径改为从设备表聚合绑定情况

为避免 gate 侧继续暴露单设备心智，出入口列表与相关视图需要改为根据 `devices.gate_id` 聚合设备，而不是读取 `gates.device_id`。本期规范层面要求：

- 出入口配置不再承担设备绑定入口。
- 出入口视图展示“已绑定设备数量/摘要”，而不是单个绑定设备。
- 删除出入口时，需要解绑该出入口下所有设备。

这样可以让 Issue #17 的绑定规则与既有 parking-lot-gates 规范保持一致。

## Risks / Trade-offs

- [遗留字段仍存在] `gates.device_id` 不会在本期删除，代码里仍可能有旧路径读取它 → 在实现阶段明确把读路径切到 `devices.gate_id`，并停止在新流程中依赖该字段。
- [平台管理员查询链路扩展] 车场列表新增 `tenant_id` 过滤后，需要谨慎处理权限，避免 tenant_admin 越权查询 → 服务层仅允许 `platform_admin` 传入该过滤条件，tenant_admin 一律忽略外部传入的 tenant_id。
- [直接改绑的事务一致性] active 设备跨租户/跨出入口改绑时，如果校验与更新分离，可能造成容量竞争 → 使用单事务完成计数与更新，并在失败时返回明确业务错误。
- [解绑状态决策与用户预期不一致] issue 中对解绑后状态仍有备注歧义 → 在 proposal/spec/design 中统一为“回到 pending + 平台池”，实现前无需再做二次猜测。

## Migration Plan

1. 先增加设备绑定/解绑服务与 API，以及所需的权限和容量校验。
2. 扩展车场列表过滤能力，满足平台管理员绑定弹窗的选项加载。
3. 调整设备管理页，接入绑定弹窗与解绑按钮。
4. 修改出入口相关读路径和规范，停止把设备绑定入口放在 gate 创建/编辑流程。
5. 保留 `gates.device_id` 字段但不再作为新逻辑真源；后续若 gate 相关功能完全迁移，再单独提案移除该遗留字段。

回滚策略：

- 若前端交互出现问题，可先回滚设备页入口而保留后端 API。
- 若后端逻辑异常，可整体回滚 bind/unbind 路由与服务实现，不涉及数据库破坏性迁移。

## Open Questions

- 出入口列表在本期前端中展示“绑定设备数量”还是“最多显示前 3 台设备摘要”，属于表现层选择；不影响本提案的后端模型和接口契约。
